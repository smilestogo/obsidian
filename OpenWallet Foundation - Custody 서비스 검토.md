## 4. Custody 서비스 개요

### 4.1 개념

**Digital Asset Custody**
- 가상자산, 증권형토큰(STO), 디지털 증권의 안전한 보관
- 전통 금융의 예탁결제원 역할을 디지털 자산에 적용
- Private Key 관리가 핵심

### 4.2 전통적 Custody vs OpenWallet Custody

| 구분 | 전통적 Custody | OpenWallet Custody |
|------|--------------|-------------------|
| 보관 대상 | 주식, 채권 (전자증서) | 가상자산, 토큰, NFT |
| 소유권 증명 | 예탁원 장부 | 블록체인 + Private Key |
| 기술 스택 | 중앙화 DB | 분산원장 + 지갑 |
| 핵심 리스크 | 시스템 장애 | Private Key 유출 |
| 규제 | 예탁결제원법 | 가상자산법 + 자본시장법 |

## 6. OpenWallet 도입 시 핵심 고려사항

### 6.1 화웨이 리스크 관리 (최우선)

#### Critical Issue
- Custody는 Private Key를 다루는 가장 민감한 영역
- 화웨이 모듈이 Key 관리에 관여 시 치명적

#### 대응 방안

**A. 모듈 분리 전략**
```
[OpenWallet 아키텍처]
├── UI/UX Layer (화웨이 참여 가능)
├── Business Logic Layer (화웨이 참여 제한)
└── Key Management Layer (화웨이 완전 배제) ← 핵심!
```

**B. 대체 솔루션 병행**
- OpenWallet은 프론트엔드만 활용
- Key 관리는 검증된 별도 솔루션 사용
  - Fireblocks (이스라엘)
  - BitGo (미국)
  - Ledger Enterprise (프랑스)
  - 국내: Dunamu (두나무), DSRV Labs

**C. 자체 개발 우선**
- 핵심 Key 관리 모듈 100% 자체 개발
- OpenWallet은 참고 표준으로만 활용
- 코스콤, 금융결제원 협업 검토

### 6.2 기술 아키텍처 설계

#### Multi-Sig (다중서명) 필수
```
고객 출금 요청
→ 내부 승인 1 (운영팀)
→ 내부 승인 2 (보안팀)  
→ 고객 승인 (2FA/생체인증)
→ 블록체인 전송
```

#### HSM (Hardware Security Module) 활용
- Private Key를 물리적으로 격리된 하드웨어에 보관
- FIPS 140-2 Level 3 이상 인증
- 국내 데이터센터 내 물리적 위치

#### Hot/Warm/Cold Wallet 구조
```
Hot Wallet (온라인)
└── 일일 출금 한도 내 자산만 (예: 전체의 2%)
└── 즉시 출금 처리용

Warm Wallet (준오프라인)
└── 주간 운영 자금 (예: 전체의 8%)
└── 수 시간 내 출금 가능

Cold Wallet (오프라인)
└── 대부분 자산 (예: 전체의 90%)
└── 물리적으로 격리된 장소
└── Multi-sig + 매뉴얼 프로세스
```

### 6.3 보안 체계

**Zero-Trust Architecture**
- 내부자 접근도 의심
- 모든 접근 로그 블록체인 기록
- 실시간 이상 거래 탐지

**DDoS/해킹 대응**
- Rate Limiting (출금 횟수 제한)
- Geo-Fencing (허용 지역 제한)
- Velocity Check (평소 패턴 이탈 시 차단)

**개인정보보호**
- Private Key 절대 평문 저장 금지
- 고객별 격리된 Key 관리
- ISMS-P, ISO 27001 인증 필수

### 6.4 법적 리스크 관리

**명확한 법적 지위 확보 필요**

Q: Custody 사업자는 누구인가?
- 증권사? → 자본시장법 적용
- VASP? → 가상자산법 적용
- 신탁업? → 신탁업법 적용
- 전자금융업? → 전자금융거래법 적용

**권고**: 금융위에 사전 질의 후 진행

**자산 분리 보관 입증**
- 고객 자산 ≠ 회사 자산
- 회사 파산 시에도 고객 자산 보호
- 정기 Proof of Reserve (지급여력 증명)

**배상책임 보험**
- 해킹/유출 시 고객 손실 보상
- 국내 보험사 상품 부재 → 해외 보험 검토

### 6.5 운영 프로세스

**Key Ceremony (키 생성 의식)**
```
1. 물리적 격리 공간에서 진행
2. 여러 담당자 입회 (최소 3명)
3. 전 과정 영상 기록
4. Key를 조각내어 분산 보관 (Shamir's Secret Sharing)
5. 복원 시에도 다수 입회 필요
```

**정기 감사**
- 내부 감사: 월 1회
- 외부 감사: 분기 1회
- Proof of Reserve 공개: 실시간 또는 일 1회

**재해복구(DR)**
- 주 데이터센터와 DR 센터 분리
- Cold Wallet 물리적 이중화
- 복구 시나리오 정기 훈련 (분기 1회)

### 6.6 비즈니스 모델

**수익 구조**
```
1. 보관 수수료: 연 0.1~0.5% (자산 규모 기준)
2. 입출금 수수료: 건당 또는 금액비례
3. 스테이킹 수익 배분: 50~90% 고객 환원
4. API 연동 수수료: B2B 거래소/증권사 대상
```

**목표 고객**
- **B2C**: 고액 자산가 (HNWI)
- **B2B**: 
  - 소형 가상자산 거래소
  - 증권사 (STO 발행사)
  - 자산운용사 (가상자산 펀드)
  - 기업 재무 (비트코인 보유 기업)

### 6.7 경쟁 환경

**국내 플레이어**
- 두나무 (업비트): 자체 Custody 운영
- 코빗, 빗썸: 자체 인프라
- 코스콤: 증권형토큰 Custody 준비 중
- 금융결제원: CBDC 인프라 구축 경험

**해외 선도 업체**
- Coinbase Custody: $300B 보관
- Fireblocks: MPC 기술
- BitGo: 규제 준수
- Anchorage Digital: 미국 연방 은행 인가

**차별화 포인트**
- 증권형토큰 + 가상자산 통합 Custody
- 전통 증권사 기반 신뢰도
- 국내 규제 최적화
- PowerBASE 연동 (증권 + 토큰 통합 관리)

### NO-GO 조건 ❌

다음 경우 추진 보류:
1. 화웨이 모듈 배제 불가능
2. 규제 불확실성 해소 안됨
3. 경영진 중장기 투자 의지 부족
4. 보안 사고 대응 체계 미흡
5. 기존 사업에 부정적 영향

### 대안 전략

**Option A: 자체 개발 (Full In-house)**
- OpenWallet 완전 배제
- 처음부터 자체 설계
- 시간/비용 多, 리스크 최소

**Option B: 글로벌 솔루션 도입**
- Fireblocks, BitGo 등
- 빠른 출시, 높은 신뢰도
- 라이선스 비용 발생

**Option C: 코스콤/금결원 협업**
- 국내 공공 인프라 활용
- 규제 리스크 최소화
- 독자 경쟁력 제한

## 12. 핵심 결론

### OpenWallet Foundation의 이중성

**긍정적 측면**
- 글로벌 표준 개발 노력
- Google, Microsoft, Visa, Mastercard 참여
- 오픈소스 투명성

**우려 측면**
- 화웨이의 깊은 관여 (Premier + 이사회)
- 미중 기술 전쟁의 전장
- 한국 기업 부재로 입장 반영 어려움

### Custody 사업의 핵심은 신뢰

- 기술보다 중요한 것은 "안전하게 보관한다"는 믿음
- 화웨이 참여는 이 신뢰를 훼손할 수 있음
- OpenWallet은 참고만, 핵심은 자체 개발 권고

### 정기우님 본부의 강점 활용

1. **금융투자 IT 경험**: 증권 시스템 운영 노하우
2. **PowerBASE 인프라**: 기존 고객사 활용 가능
3. **규제 대응 역량**: 금융권 규제 경험
4. **차별화 포인트**: 증권 + 토큰 통합 Custody
